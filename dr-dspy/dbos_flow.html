HumanEval eval-only — DBOS control flow

Two scripts, one shared durable pipeline, read left → right: submit ▸ generate ▸ score, over DBOS workflows & steps and two Postgres-backed queues. The two rows below are stacked & column-aligned so you can compare them directly — they are identical everywhere except stage 4 (Direct = 1 LLM call, Enc-Dec = 2 sequential calls) and the generation-queue concurrency.

scripts: humaneval_dspy_eval_only_dbos_v0.py · …encdec_dbos_v0.py impl: humaneval_direct_dbos.py · humaneval_encdec_dbos.py orchestrator: DBOS + Postgres · llm: OpenRouter via dspy

Legend · DBOS constructs & touchpoints

WORKFLOW@DBOS.workflow — durable orchestration; every step result is checkpointed, so it recovers after a crash.
DBread/write to the app Postgres (experiments / predictions / batch_operations).
STEP@DBOS.step — durable unit of work & retry boundary (most max_attempts=3).
SYSread/write to the DBOS system DB (dbos.* workflow + queue state).
QUEUEDBOS queue with a worker_concurrency cap; rows live in the system DB.
LLMoutbound OpenRouter request (chat.completions.create).
PROCESSOS-level worker process / child subprocess.
PROCsubprocess.run — child Python that executes the generated code.
differsthis stage is not identical between the two pipelines.
QDBOS.enqueue_workflow — place a workflow on a queue (idempotent).

PROCESS Worker process (worker)

run_worker_command() — one long-lived process, blocks until halt:

  • raise_open_file_limit() — FD budget = scoring_concurrency × fds_per_worker
  • configure_dbos_runtime() — register queues w/ concurrency, reconcile via UPDATE dbos.queues SYS, launch DBOS, listen on selected queues
  • configure_openrouter_client() — per-process cached httpx.Client+OpenAI; pool sized from gen concurrency LLM pool
  • Postgres pool + start_worker_monitor() thread (polls predictions + dbos.*)

⚠ No persistent scoring pool. Scoring parallelism = scoring-queue concurrency; each score step spawns short-lived subprocess.run children (one per top-level fn name) and reaps them.

Per-queue worker_concurrency (bars relative to 200). Generation is network-bound; scoring is CPU/process-bound, so it's kept low.

defaults per script
  • direct: 10 samples · temps {0.0} · 1 rep · 16 models · 1000 tok
  • enc-dec: 164 tasks · 7 budget ratios · 3 reps · self-paired models · 2000 tok
WORKFLOW enqueue-scores

enqueue_scores_dispatcher_workflow → loops enqueue_scores_batch_step:

  • fetch_scoreable_prediction_ids() (generated, not yet scored) DB
  • re-enqueue score_prediction_workflow + mark queued Q DB
  • completion: PROCESSED_TOTAL_OR_EMPTY_BATCH

Backfills scoring if the in-workflow enqueue was missed.

WORKFLOW repair

repair_dispatcher_workflow → repair_batch_step → apply_repair_batch():

  • find failed/stuck rows, reset_generation_errors_for_retry() DB
  • re-enqueue generation and/or scoring w/ fresh retry token Q SYS
  • separate generation_limit / scoring_limit; drain-to-empty

--apply launches it; dry-run only reports counts.

DB App Postgres
  • *_eval_experiments — one row per experiment
  • *_eval_predictions — one row per prediction; generation_status + scoring_status state machines, outputs, usage/cost, score, metrics. PK = prediction_id
  • dr_dspy_batch_operations — dispatcher progress + spec (PK operation_kind, operation_key)
SYS DBOS system DB
  • dbos.* — workflow status + step checkpoints, queue tables + worker_concurrency
  • separate connection (DBOS_SYSTEM_DATABASE_URL)
  • this is what makes the pipeline resumable: kill a worker mid-flight and DBOS recovers each workflow, skipping checkpointed steps
